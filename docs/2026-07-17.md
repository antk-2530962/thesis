# Part I — Per-run metadata & the queryable index

## 1. The problem

Run folders (`src/models/<dataset>/<architecture>/<timestamp>/`) documented
each run well for a *human reader* (`README.md`, `data.md`) but not for a
*query*: finding the best N runs for a (dataset, architecture, subset)
combination meant opening READMEs one by one. The structured fragments that
did exist were inconsistent — the two older runs had `cache/eval_summary.json`
but no hyperparameter record, the three 2026-07-16 runs had `cache/hyp.json`
but no eval numbers.

## 2. `metadata.json` — the per-run record

One `metadata.json` at each run folder root (schema_version 1): identity
(`run_id`, `dataset`, `architecture`, `model_name`, `streaming`), input
(`subset`, `n_landmarks`, `coords`, `feature_dim`), scale (`n_params`),
protocol (`split`, `n_val`, `n_classes`), provenance (`trained_date`,
`training_source`, `epochs`, `best_epoch`), results, and a nested
`hyperparameters` dict.

Results carry an explicit **`eval_status`** so training-loop numbers can never
masquerade as leaderboard numbers:

| field | source | eval_status |
|---|---|---|
| `train_val_acc` | training loop (`gru_latest.pt` history) | written at `"pending"` |
| `overall_accuracy`, `macro_accuracy`, `median_class_accuracy`, `n_classes_below_50pct` | `scripts/eval_gru.py` on the canonical split | promotes to `"canonical"` |

Backfilled 2026-07-17 for all 5 existing gislr/gru runs (the two pre-`hyp.json`
runs reconstructed from their READMEs + `eval_summary.json`).

## 3. Lifecycle — who writes what

1. **Training notebook** (`gislr.1.model.gru.ipynb` §7 run-docs cell) writes
   `metadata.json` alongside the `data.md`/`README.md` drafts —
   `eval_status: "pending"`, `best_epoch` from the checkpoint history.
   Re-running the cell preserves canonical-eval fields already filled in.
2. **`scripts/eval_gru.py`** (canonical per-class eval) updates the run's
   `metadata.json` in place and flips `eval_status` to `"canonical"`.
3. **`scripts/build_model_index.py`** (new; runs with CWD anywhere, like
   `eval_gru.py`) flattens every `metadata.json` into
   **`src/models/index.csv`** (committed; hyperparameters as `hyp_*` columns)
   and answers filter queries directly:

```bash
.venv/Scripts/python.exe scripts/build_model_index.py                       # rebuild + full leaderboard
.venv/Scripts/python.exe scripts/build_model_index.py --dataset gislr --arch gru --top 3
.venv/Scripts/python.exe scripts/build_model_index.py --subset ME_126
```

The index's sort column `val_acc` = canonical `overall_accuracy` when present,
else `train_val_acc`; rows missing `metadata.json` are warned about, never
silently dropped. Rebuild after each training run or canonical eval.

# Part II — Fresh-run-folder policy

## 5. Before / after

`resolve_run_dir` in `gislr.1.model.gru.ipynb` used a per-subset pointer file
(`cache/gru_runs/<tag>.txt`) that made *any* re-run land in the existing
folder; `train_subset` then skipped completed runs. Retraining under the same
conditions was therefore impossible without manually deleting the pointer.

Now the pointer is followed **only while the pointed-to run is incomplete**
(checkpoint epoch < configured epochs — auto-resume after an interrupt is
unchanged, per the §1.2 resumability convention). A completed run is never
reused: every new training start creates a fresh
`models/gislr/gru/<YYYYMMDD-HHMMSS>/` folder (timestamp = training start) and
repoints the pointer. Reproducibility re-runs like 20260716-213901 now happen
by just re-running the training cell.

**Consequence to be aware of:** the training cell (§5 of the notebook) is no
longer idempotent — re-running it retrains *every* subset in `RUN_SUBSETS`
(~25 GPU-min each). Trim `RUN_SUBSETS` when that isn't wanted (noted in the
cell banner).
